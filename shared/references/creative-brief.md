# Creative Brief 渐进式规则

Creative Brief 是 Agent 的渐进式工作记忆，不是用户必填表单，不是 Base 字段，也不是每一步都要填满的强 schema。

它只服务创意生成和跨 skill 心智继承，不服务机械流程。环境检查、ID 回查、配置页保存、状态查询、图片压缩、打包、Base 写入、附件续传等 Fast Path 不需要读取或补全 Creative Brief。

## 核心框架

创意生成前尽量建立这条理解链：

```text
功能事实 → 目标用户 → 具体场景 → 用户阻碍/情绪 → 产品动作 → 可感知变化
```

这条链不是给用户看的，也不是固定输出格式。它用于防止方向、文案、图片和迭代只靠漂亮词，忽略产品事实和真实用户处境。

## 三档理解

| 档位 | 适用入口 | 处理 |
|---|---|---|
| Full | 从方向开始，或用户给了功能、卖点、人群、场景 | 尽量补齐完整理解链 |
| Lite | 从 C-ID、临时文案、用户粘贴文案或图片需求开始 | 至少保留锚点、目标用户、已知产品动作和缺失项 |
| Unknown | 只有旧图、模糊口头需求或外部素材，无法判断功能事实 | 标记缺失；影响卖点、文案或 prompt 时追问 |

能 Full 就 Full；不能 Full 就 Lite；关键功能事实缺失才进入 Unknown。不要为了凑 Full Brief 打断正常流程。

## 关键门禁

必须尽量明确：

- 目标用户：默认学生；用户明确指定家长、老师、校方或投放负责人时覆盖。
- 锚点来源：D-ID、C-ID、G-ID、临时文案、用户上传图、用户资料或当前对话产物。
- 产品动作：洋葱具体做了什么，例如一拍定位、分步解析、继续追问、诊断分班、定制计划、30 分钟课堂、AI+真人辅导。

产品动作不清楚，并且要生成方向、文案、图片 prompt、新卖点或换功能时，先查知识库；仍不清楚就追问。不能为了共鸣编功能。

可以推断：

- 具体场景。
- 用户阻碍/情绪。
- 可感知变化。

推断必须合理，不能夸大焦虑，不能写绝对承诺，不能跨功能迁移场景。

## 学生视角

面向学生时，默认语气活泼、口语化、符合学生视角，让学生看到后有“这就是我”的共鸣。

学生视角不是强制第一人称。可以写“我”，也可以不用“我”；关键是场景、情绪和语言来自学生本人，而不是家长腔、品牌说明书或销售话术。

运营认可文案只学习共鸣强度、口语化程度、产品动作表达和可感知变化，不照搬固定句式，不把私有飞书原文案整批复制进插件。

## 功能边界

不同功能事实不能互相迁移：

- 不要把拍题精学的作业卡题场景套到私教班、同步课或其它功能。
- 不要把私教班的诊断分班、AI+真人辅导写到拍题精学。
- 不要把拍题精学的一拍解析、分步讲解、继续追问写成私教班独有能力，除非用户明确要求组合卖点且业务资料支持。
- NPC 老师、动画讲透、私教老师、AI+真人辅导等词，只有在对应功能或用户 brief 明确出现时才能用。

## 各 skill 用法

| Skill | 怎么用 |
|---|---|
| `onion-direction` | 生成方向前尽量建立 Full/Lite Brief；字段 1/2/3 分别承接场景问题、产品动作/差异解法、可感知变化 |
| `onion-copy` | 从方向、文案或知识库提炼学生场景、产品动作和可感知变化；产品动作不清楚时先查知识库或追问 |
| `onion-image` | prompt 视觉化 Creative Brief，让画面承接用户场景、产品动作和可感知变化；配置、压缩、打包、Base 写入、状态查询不需要 Creative Brief |
| `onion-image-iterate` | G-ID 扩同类时继承原图组的 Brief/metadata；纯视觉微调不强制补全功能事实；换卖点、换功能、换文案时重新补 Brief |
