# 状态摘要查询

> onion-help skill 段 2️⃣ + 段 3️⃣ 调用。拉 4 张表的 status 计数 + 给轻量推荐。

---

## 一、调 lark-cli 拉 4 张表的 status 计数

### 推荐做法：用 `+data-query` 做分组聚合

lark-base skill 文档里说 `+data-query` 适合做"分组统计、SUM / AVG / COUNT 等聚合分析"。**这正是我们需要的**——按 status 分组数 records。

但 `+data-query` 要求 **当前用户是 Base 管理员（FA 权限）**。如果不是 FA，会返回权限错误。

**Fallback 方案**：用 `+record-list --limit 200 --format json` 拉每张表全部记录，AI 在内存里按 status 分组数 count。Base 行数不大（每表 < 200 条）的时候这个方案够用。

### 4 张表 + status 字段对照

| 表 | table_id | status 字段名 | status 值 |
|---|---|---|---|
| directions | tblLWPSHrZT95oy7 | `状态` | 待用 / 已用 / 废弃 |
| copies | tblFdwXSbjANQjlh | `状态` | 待用 / 已用 / 废弃 |
| image_groups | tblGpuukciptN3PP | `状态` | 待用 / 已用 / 废弃 |
| feedbacks | tblsPpNNcNH5KXoZ | `处置状态` | 待审 / 已转SKILL.md规则 / 已采纳到下次迭代 / 已驳回 |

### 命令模板（Fallback 方案，AI 内存分组）

```bash
BASE=WIoGb0ksnaREvJsPtQCcW8Lsnfg

# directions
lark-cli base +record-list --base-token $BASE \
  --table-id tblLWPSHrZT95oy7 --field-id 状态 --limit 200 --format json --as user 2>/dev/null \
  | jq -r '.data.data[]?[0] // "未填"' | sort | uniq -c

# copies
lark-cli base +record-list --base-token $BASE \
  --table-id tblFdwXSbjANQjlh --field-id 状态 --limit 200 --format json --as user 2>/dev/null \
  | jq -r '.data.data[]?[0] // "未填"' | sort | uniq -c

# image_groups
lark-cli base +record-list --base-token $BASE \
  --table-id tblGpuukciptN3PP --field-id 状态 --limit 200 --format json --as user 2>/dev/null \
  | jq -r '.data.data[]?[0] // "未填"' | sort | uniq -c

# feedbacks（按"处置状态"分组）
lark-cli base +record-list --base-token $BASE \
  --table-id tblsPpNNcNH5KXoZ --field-id 处置状态 --limit 200 --format json --as user 2>/dev/null \
  | jq -r '.data.data[]?[0] // "未填"' | sort | uniq -c
```

每个命令解析后得到类似：
```
   8 待用
  12 已用
   3 废弃
```

## 二、输出格式

```
📊 Base 当前状态（截至 2026-05-17 14:30）

  directions   待用 8  / 已用 12 / 废弃 3  / 共 23 条
  copies       待用 12 / 已用 35 / 废弃 5  / 共 52 条
  image_groups 待用 5  / 已用 18 / 废弃 2  / 共 25 套
  feedbacks    ⚠️ 待审 3 / 已转规则 4 / 已采纳 0 / 已驳回 1
```

显示规则：
- 各表 status 按 `待用 / 已用 / 废弃` 固定顺序
- 「共 N 条」是三者之和（验算用，万一某个 status 数据异常能看出来）
- feedbacks "待审 ≥ 1 条" → 加 ⚠️ emoji 提示维护人
- 数字按表格对齐（不对齐也行，可读性优先）

## 三、轻量推荐下一步

按下面规则**列 1-3 条可选**，**不强制顺序**：

| 触发条件 | 推荐 | 说明 |
|---|---|---|
| `directions.待用 > 0` | "调 onion-copy 给这些方向出文案" | 上游有料，下游可消化 |
| `copies.待用 > 0` | "调 onion-image 出图" | 同上 |
| `image_groups.已用 ≥ 5` 且最近 7 天有更新 | "调 onion-image-iterate 基于跑量好的图扩同类" | 有"经过验证"的好图就扩 |
| `feedbacks.待审 > 0` | "抽时间在 Base 里处理 feedbacks 待审项" | 不影响核心流程但建议定期清 |
| 所有 status 都是 0（全新 Base）| "可以从 onion-direction 出第一批方向开始" | 新 Base 的起点 |
| 上面都不命中 | "看着办，可调任一 skill" | 不强行推荐 |

### 输出格式

```
💡 下一步你可以：

  • directions 有 8 条待用 → 调 onion-copy 出文案
  • copies 有 12 条待用 → 调 onion-image 出图
  • feedbacks 待审 ⚠️ 3 条 → 抽时间在 Base 里处理
  
  也可以直接调 onion-image-iterate 扩已有图组（不需要前置流程）。
```

末尾加一句"也可以直接调 onion-image-iterate 扩同类"作为**提示非线性流程也支持**，避免运营被流程绑死。

---

## 四、推荐之外不写

❌ **不写**："你必须先调 onion-copy 才能调 onion-image"
❌ **不写**："建议你优先调 onion-direction 因为它最简单"
❌ **不写**："今天先别调 onion-image-iterate，因为没有好图"

→ **不评判 / 不强制 / 不限制**。运营自己看完状态后决定。
