# onion-direction

## 任务目标

把用户的产品功能、节点、人群或样本材料转成可进入投放生产链路的方向卡。方向卡是上游产物，下游会被 `onion-copy` 和 `onion-image` 引用。

成功的方向卡不是功能介绍，而是一个能被拍成 3 秒画面的投放切入点：具体人群、具体场景、具体痛点、差异化解法和可感知奇效都成立。

## 输入契约

进入方向生成前，先按 `../../shared/references/input-envelope.md` 整理 Input Envelope。本 skill 只校验方向切片：`business.feature`、`business.selling_points`、`business.stage`、`business.stage_source`、方向原料、方向候选和方向写入策略。用户 brief、调研、访谈、案例先放在 `raw_materials`，由本 skill 提炼成结构化方向字段。

方向生成属于 Creative Path。生成前按 `../../shared/references/creative-brief.md` 尽量建立 Full/Lite Creative Brief：先理解功能事实、目标用户、具体场景、用户阻碍/情绪、产品动作和可感知变化；缺关键功能事实时先查知识库或追问。不要把拍题精学的作业卡题场景套到私教班、同步课或其它功能。

首启环境门禁：开始前先轻量检查 `~/.onion-ad/usage-state.json`、`~/.onion-ad/setup-status.json` 和 `~/.onion-ad/update-status.json`。缺本地使用记录或更新记录时先转 `onion-help` 运行 `setup_wizard.py ensure`；该命令使用 24 小时缓存检查插件版本，安全时才自动 fast-forward，设置 `ONION_PLUGIN_AUTO_UPDATE=0` 可关闭。纯候选生成可以先继续，但只要用户要写 Base、回查 D-XXX 或使用飞书表格，环境缺失、状态未知或版本检查提示风险时先环境检查。不要等到写入时才发现 lark-cli、Base token 或插件版本问题。

## 入口门禁

本 skill 只处理方向卡。明确要求出文案时转 `onion-copy`；明确要求出图时不能直接使用方向字段生图，先转 `onion-copy` 形成或确认文案；明确给 G-XXX 或旧广告图扩同类时转 `onion-image-iterate`。

越界时停止当前方向流程，只整理 handoff 所需的方向锚点和缺失项，不在本 skill 里补写文案、配置图片或调用生图。

关键输入是 `功能 + 卖点`。缺 `适配阶段` 不反问；它由当前日期、用户投放排期或 brief 里的时间节点推断，并写入 `stage_source=user|time_inferred|default`。

必填输入：

- `功能`：从 `../../shared/knowledge/卖点库.md` 的功能矩阵理解真实产品能力；写入 Base 时对齐当前表选项（常见为拍题精学 / 同步课 / 总复习 / 学情报告 / AI私教动画课 / AI定制班 / 洋葱私教班 / 错题本 / 其他，新增功能先确认选项或映射）。
- `卖点`：用户可直接给核心利益点，也可说“卖点你来定 / 按当前节点推荐”；后者从功能、当前时间节点和 `../../shared/knowledge/卖点库.md` 选择 1-3 个互补卖点。
- Base 的 `卖点` 字段是 text，不是选项。内部可以用 `business.selling_points` 数组做推理和多样性控制；写入 directions 表时必须合并成文本，例如 `识别准确；AI+名师校准`，不要传 JSON 数组。

可推断输入：

- `基准方向`：用户给 D-XXX / directions record_id 时先回查 Base；用户粘完整方向时直接用上下文。
- `目标人群`：用户给了年级、学习状态、场景或样本时可推断；否则可先用功能和节点生成一个合理细分。
- `适配阶段`：日常学习 / P0-寒假 / P0-开学 / 期中前 / 期末前 / 暑期 / 其他。用户指定时 `stage_source=user`；未指定时按当前日期或 brief 时间节点推断，命中则 `stage_source=time_inferred`；完全没有节点信号时默认日常学习，`stage_source=default`。
- `时间节点语义`：用户给日期、寒暑假、开学、期中、期末、月考、备考等信号时，推断学生当前状态和更适合的场景任务；用户没给节点时，也要按当前日期推断最近的学习任务。
- `数量`：用户没说时默认 3 条；仿写没说时默认 2 条。

展示给用户和写入 Base 时，每条方向都必须具备 `素材方向`、`功能`、`卖点`、`目标人群`、`适配阶段`、三段方向字段和 `状态=待用`。候选方向按 `../../shared/references/response-format.md` 固定排版，必须输出为一张 9 列 Markdown 表格：`编号 / 素材方向 / 功能 / 卖点 / 目标人群 / 适配阶段 / 具体场景问题 / 惊艳解法 / 场景奇效`。9 列都必须出现，不能写省略号，不能写“同上”，不能只给摘要；表格后只补一句操作提示，不展示后续图片建议、内部切入角度或自评分。不要传 `方向ID`、创建时间、创建人、最后更新时间；这些由飞书根据当前 `--as user` 账号自动记录。

交给下游 `onion-copy` 时，必须形成结构化 handoff：方向 record_id / D-XXX、方向字段、是否已写 Base、缺失的 `渠道` 或 `图片形式`。不要只说“已选第二个方向，去出文案”。

版本策略：方向修改默认创建新版，不覆盖原记录。用户说“改 D-XXX / 更新 D-XXX / 替换 D-XXX”时，先回查原方向并展示修改版；用户确认后用 `write_record.py` 新建一条新版方向。原方向默认不动；只有用户明确说废弃原记录时，才用 `update_status.py` 把原方向状态改为 `废弃`。当前 skill 不做原记录字段覆盖，不能假装已经覆盖更新。

## 推断原则

先判断模式：

| 信号 | 模式 | 处理 |
|---|---|---|
| 用户粘了完整方向并说类似、同款、仿照、换说法 | 仿写 | 保留结构和语气，换新角度；默认不主动入库 |
| 用户给 D-XXX 并说扩、再来、类似 | 扩方向 | 用 `../../shared/scripts/lookup_record.py` 回查原方向，保留核心人群/功能，可换卖点、场景或表达 |
| 用户给 D-XXX 并说改、更新、替换 | 改方向 | 回查原方向，先展示修改版；确认后默认创建新版，不覆盖原记录 |
| 用户给功能、卖点、人群、数量等明确 brief | 新建 | 不多问，直接生成；适配阶段按当前日期或用户节点推断 |
| 用户给调研/访谈/复盘材料 | 提炼 | 从材料里提炼场景、人群和痛点，再生成方向 |
| 用户只说“选择方向 N / 选第 N 条” | 选择确认 | 先把被选方向入库；入库后必须问下一步：是否基于该方向出文案、继续改方向，或先停在方向库 |
| 用户选择多个方向 | 多选确认 | 先确认这些方向都要入库；如要继续出文案，必须问是每个方向都出，还是只选其中一个方向出 |
| 用户只说“出方向” | 信息缺失 | 只问功能和卖点；用户说卖点你来定时，按当前时间节点和卖点库推荐 |

完整 brief 不反问。用户没给目标人群但功能和卖点足够明确时，可以自行选择一个细分人群，并在输出里自然体现，不要用确认流程拖慢创作。缺 `适配阶段` 不反问；按当前日期推断并在候选方向里写明“按当前日期推断”或“默认日常学习”。

多条方向要互补。新建创意时，任意两条至少在卖点、场景、人群、切入角度中有 2 维不同；3 条及以上至少覆盖 3 类切入角度，5 条及以上至少覆盖 4 类，同一角度最多 2 条。

时间节点要影响方向内容。不要只把“期末前/暑期/开学季”填进 `适配阶段`；字段 4/5/6 也要体现对应的学习状态、任务和解法。

方向字段 1/2/3 要承接 Creative Brief：字段 1 写用户具体场景和阻碍；字段 2 写产品动作或差异化解法；字段 3 写可感知变化。Creative Brief 不完整时不要硬凑，优先保证功能事实真实。

## 反问原则

只问阻塞项：

- 缺功能时，问用户选哪个功能。
- 缺卖点时，问用户要打哪个核心卖点；如果用户希望 Agent 自选，就按当前时间节点从卖点库推荐。
- 缺数量时不问，按默认值。
- 仿写样本足够完整时不问，先产出。

有交互 UI 时一次性收集缺失选项；没有 UI 时用一句话问清关键缺口。具体运行时适配见 `../../shared/references/runtime-adapters.md`。

## 硬约束

- 方向卡必须使用 `references/字段与生成规则.md` 里的 6 字段语义。
- 候选方向展示时必须显式显示 `适配阶段`；不要只在内部字段里保留。
- 候选方向必须按 `../../shared/references/response-format.md` 输出为一张 9 列表格；不能写省略号，不能写“同上”，不能展示后续图片建议或内部切入角度。
- `素材方向` 要像可投放主题，不要写成产品功能名。
- 字段 1 写具体场景里的具体问题；字段 2 写差异化解法；字段 3 写同一场景下的可感知变化。
- 不写包过、保证、提分 N 分、100%、唯一、第一等承诺词。
- 不写洋葱 APP 没有的能力。
- 广告主语默认是洋葱学园 APP 学生端，不把教师版、校园版、合作硬件或其他产品线写成主角。
- 迭代过程不写 Base。用户明确确认“OK / 满意 / 就这样 / 存进去”时写入；用户说“选择第 N 条 / 方向一 / 就用这个”视为确认采纳该方向，先只把被选中的方向写入 Base。
- 改方向默认创建新版，不覆盖原记录；用户明确说废弃原记录时，先确认后把原方向状态改为 `废弃`。用户明确要求“覆盖原记录”时，说明当前不支持覆盖字段，只能创建新版并保留历史。
- 如果用户只是选择方向，没有明确说“出文案 / 继续生成文案”，入库后必须问下一步，不能直接结束也不能擅自出文案。
- 如果用户选择方向并明确要求出文案，入库后进入 `onion-copy`；若缺渠道或图片形式，由 `onion-copy` 继续反问。
- 如果用户一次选择多个方向并要求出文案，必须先问“每个方向各出一批，还是只基于其中一个方向出”，不要默认只用第一个。
- 仿写默认先展示再问是否入库，避免污染方向库。
- 出方向不依赖 `LAOZHANG_API_KEY`，不能因为缺老张 API key 阻塞。

## 工具调用

用户给 D-XXX 但上下文没有方向内容时，先用 `../../shared/scripts/lookup_record.py --id D-XXX` 回查；拿到 record_id 后再决定扩写、展示或确认更新。

用户确认入库后，用 `../../shared/scripts/write_record.py` 写入 directions 表，table id 来自 `../../shared/base_schema.md` 或环境变量。

传给脚本的记录使用 `records[].fields` 结构，由脚本转换为当前 lark-cli 需要的 `fields/rows` payload。脚本负责 dry-run、重试和 pending 队列。

方向创建本身没有上游状态要更新；不要调用状态更新工具。只有用户明确要求废弃原方向时，才用 `../../shared/scripts/update_status.py` 把原 directions record_id 的 `状态` 改为 `废弃`。
