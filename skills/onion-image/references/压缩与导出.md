# 压缩与导出

> 用来解决“生成后怎么适配版位、压缩、写 Base”的执行判断。字段 schema 仍以 `shared/base_schema.md` 为准。

## 默认策略

| 维度 | 默认值 | 理由 |
|---|---|---|
| 生图 quality | `medium` | 投放够用，high 慢且费钱 |
| 生图 size | `config/channel-placement-rules.json` 的 `render_size` | 用户只选版位，不选比例 |
| 目标尺寸 | `target_size` | 写 Base 和本地 zip 前必须导出到目标尺寸 |
| 压缩 | 写入 Base 前压缩 | Base 存投放可用附件，避免多人共用附件过大 |
| 目标 KB | `max_file_size_kb` 优先；未知版位默认 200KB | 兼顾审核和通用可控体积 |
| Base 附件格式 | JPG | `write_image_group.py` 会把 PNG 转为压缩 JPG |
| 本地交付包 | 采纳图打 zip | 本地保留一份所有通过图片，便于运营下载、复盘或转交 |
| 原图清理 | 默认 7 天后自动清理 | 原始 PNG 大，短期可迭代，长期占磁盘 |

Base 的 `图1/图2/图3` 上传按目标尺寸导出并压缩后的投放附件，不上传 render.py 原始 PNG。原始 PNG 只保留在本地临时目录。

本地 zip 由 `package_accepted_images.py` 从 `image-selection-result.json` 的 `accepted_schemes` 生成。zip 只包含采纳图，默认使用按目标尺寸导出的压缩 JPG，并写入 `manifest.json` 记录来源、set_id、meta、target_size 和源文件路径。

## 尺寸适配策略

| 版位状态 | 处理 |
|---|---|
| `direct` | 用 `render_size` 渲染，resize 到 `target_size`，再压缩 |
| `near` | 用 `render_size` 渲染，cover resize + center crop 到 `target_size`，再压缩 |
| `expand` | 一期置灰；后续接扩图工具后再开放 |
| `composite` | 一期置灰；后续接模板/分层合成后再开放 |
| 双图 / 三图 / 组图 | 一期置灰；后续做多图版位流程 |

不要把一个版位的成图硬裁成另一个版位。同一文案选多个版位时，每个版位应单独生成 prompt 和原始 PNG。

## 导出判断

```text
用户选一个或多个版位
  ↓
查 config/channel-placement-rules.json 得到 render_size / target_size / KB
  ↓
enabled=true
  是 → 用 render_size 调 render.py
  否 → 置灰，不进入本期生成
  ↓
用户确认采纳
  ↓
读取 image-selection-result.json
  ↓
package_accepted_images.py 按 target_size / target_kb 打包 accepted_schemes 为 zip
  ↓
调用 write_image_group.py 写入 Base，脚本负责目标尺寸导出、压缩和附件上传
  ↓
write_image_group.py best-effort 清理 /tmp/onion-ad 下超过 7 天的原始 PNG
```

## 写入 Base

用户确认采纳后调用：

```bash
PLUGIN_ROOT=/path/to/onion-toufang-plugin
python3 "$PLUGIN_ROOT/shared/scripts/write_image_group.py" ...
```

传参原则：

- 版位有明确目标尺寸：在 images 里传 `target_width` / `target_height`。
- 版位有明确文件大小上限：传 `--target-kb <上限>`，或在 metadata / images 里传 `目标KB`、`target_kb`。
- 版位没有明确上限：不传目标，脚本使用默认 200KB。
- 只有调试或维护人明确要求保留原图附件时，才传 `--no-compress`。
- `创建人`、`创建时间`、`图组ID` 等系统字段不要传。

## 本地打包

写 Base 前先打包：

```bash
python3 skills/onion-image/scripts/package_accepted_images.py \
  --selection-result /tmp/onion-ad/<request_id>/image-selection-result.json \
  --target-kb 200
```

输出：

```text
/tmp/onion-ad/<request_id>/<request_id>-accepted-images.zip
```

这个 zip 是本地留档/交付包，不写 Base。里面只放 `accepted_schemes`，拒绝和未标注的图片不打包。若 accepted scheme 的 meta / placement 含 `target_width` 和 `target_height`，zip 内图片会按目标尺寸导出。

## 原图保留与清理

原始 PNG 用于短期检查、继续迭代和 branch 参考，不能生成后立刻删。默认策略：

- 原始 PNG 保留 7 天。
- 压缩 JPG、zip、HTML、JSON 默认不自动删除。
- 每次 `write_image_group.py` 成功写入 Base 后，best-effort 调用 `cleanup_image_outputs.py` 清理 `/tmp/onion-ad` 下超过 7 天的原始 PNG。

手动预检：

```bash
python3 skills/onion-image/scripts/cleanup_image_outputs.py \
  --root /tmp/onion-ad \
  --original-retention-days 7 \
  --dry-run
```

当前量级参考：单张原始 PNG 通常约 1-2MB，压缩后约 100-200KB。50 套三图会留下约 150 张原图，可能占 150-300MB；如果每天跑多批，不自动清理会很快累积到数 GB。

## 原图分辨率

应用商店目标尺寸通常很小，例如 320×210、1280×720。生成时用 `render_size`，后期缩小相对安全；不要用过低分辨率生成再放大到大版位。

| 安全 | 不安全 |
|---|---|
| 从 1568×672 缩到 984×422 | 从 720×1280 放大到 1080×1920 |
