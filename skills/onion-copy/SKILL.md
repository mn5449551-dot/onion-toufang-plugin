# onion-copy

## 任务目标

把方向卡转成可给 `onion-image` 使用的投放文案。文案是第二层产物：上游继承方向心智，下游决定图中文字结构。

成功的文案要让人一眼知道痛点和产品动作，同时符合默认学生视角、渠道语气、图片形式和 Base 字段结构。

## 输入契约

进入文案生成前，先按 `../../shared/references/input-envelope.md` 整理 Input Envelope。本 skill 只校验文案切片：方向/文案锚点、`business.channel`、`business.image_form`、文案候选和文案写入策略。用户粘贴的参考文案、样本和口头要求可以先放在 `raw_materials`，再由本 skill 提炼成具体字段。

文案生成属于 Creative Path。生成前按 `../../shared/references/creative-brief.md` 建立“功能 × 用户”理解：当前功能是什么、目标用户是谁、用户在哪个具体场景遇到什么阻碍、洋葱用什么产品动作解决、用户能感知到什么变化。产品动作不清楚时先查知识库或追问，不能为了共鸣编功能。

首启环境门禁：开始前先轻量检查 `~/.onion-ad/usage-state.json` 和 `~/.onion-ad/setup-status.json`。缺本地使用记录时先转 `onion-help` 运行 `setup_wizard.py ensure`；纯文案候选可以先继续，但涉及 D/C ID 回查、写 Base、更新方向状态或准备进入出图时，若环境未知或未就绪，先环境检查。

## 入口门禁

本 skill 只处理投放文案。明确出方向、扩方向或改方向时转 `onion-direction`；明确用已确认文案 / C-ID / 临时文案出新图时，本 skill 只负责先确认并入库被选文案，再转 `onion-image`；明确给 G-XXX 或旧广告图扩同类、换角色、换文案时转 `onion-image-iterate`。

越界时不要在本 skill 里配置图片、生成 prompt、调用 render 或写 image_groups。只整理 copy handoff：文案字段、copy record_id / C-XXX、渠道、图片形式和缺失视觉配置。

必填输入：

- `方向或文案基准`：方向 record_id / D-XXX / C-XXX / 用户粘贴的方向卡 / 用户粘贴的方向内容 / 用户粘贴的文案样本。
- `渠道`：信息流 / 应用商店 / 学习机。
- `图片形式`：单图 / 双图 / 三图。学习机只支持单图；应用商店保留单图、双图、三图。

可推断输入：

- `关联方向`：用户给 D-XXX 时先解析成 directions record_id；用户只粘方向内容时可留空。
- `基准文案`：用户给 C-XXX 时先回查 copies；扩写时继承渠道、图片形式和原方向血缘，除非用户明确覆盖。
- `轻量扩写锚点`：用户直接粘一段标题/短句/旧文案并要求“类似、扩写、换说法”时，以当前文案作为锚点即可，不强行追问方向 ID 或回查上游。
- `数量`：用户没说时按默认规则。
- `文案类型`：从渠道、方向心智和切入角度选择；作为参考维度，不强制凑类型。

交给下游 `onion-image` 时，必须形成结构化 handoff：copy record_id / C-XXX、文案字段、渠道、图片形式、是否已写 Base、缺失的视觉配置项。不要只说“第二条文案可以出图”。

展示给用户时，按 `../../shared/references/response-format.md` 的文案候选固定排版输出。多渠道/多形式文案必须先做合法组合过滤，再输出生成范围表，最后按渠道/形式分组表格展示。仅当用户同时要求应用商店和学习机单图时，应用商店单图和学习机单图合并展示为 `应用商店 / 学习机 · 单图`；用户只要求学习机单图时，展示 `学习机 · 单图`，不强行合并应用商店；用户只要求应用商店单图时，展示 `应用商店 · 单图`，不强行合并学习机。编号全局唯一，不能在每个分组里重置。`文案类型/角度` 只用于内部多样性判断和必要 Base 字段，不在用户候选回复里展示；用户只需要看到渠道、图片形式、主标题/副标题或短句；关联方向可在表格前统一说明，不逐行塞进候选表格。

版本策略：文案修改默认创建新版，不覆盖原文案。用户说“改 C-XXX / 更新 C-XXX / 替换 C-XXX”时，先回查原文案并展示修改版；用户确认后用 `write_record.py` 新建一条新版文案，继承可信的关联方向、渠道和图片形式。原文案默认不动；只有用户明确说废弃原文案时，才用 `update_status.py` 把原文案状态改为 `废弃`。当前 skill 不做原记录字段覆盖，不能假装已经覆盖更新。

默认数量：

- 信息流单图：3 套。
- 应用商店：用户没指定形式时，单图 1 套、双图 1 套、三图 1 套；用户指定形式时按指定形式出。
- 学习机：只出单图，默认 1 套；用户要求学习机的双图或三图时，先说明学习机只支持单图，再问是否改成学习机单图或应用商店多图。
- 仿写扩展：默认 2 套。

## 推断原则

先判断模式：

| 信号 | 模式 | 处理 |
|---|---|---|
| 用户给方向 ID / 方向内容 + 渠道 + 图片形式 | 新建 | 直接出文案 |
| 用户说“选择第 N 条方向 / 方向一”并要求出文案 | 选择续跑 | 若该方向尚未入库，先写入被选中的方向；再检查渠道和图片形式，缺任一项就问，不默认信息流或单图 |
| 用户给 D-XXX 但上下文没有方向 | 断点续跑 | 用 `../../shared/scripts/lookup_record.py` 回查 directions 后继续 |
| 用户给 C-XXX 并说扩、类似、再来 | 扩文案 | 回查 copies，可继承关联方向、渠道、图片形式和原结构，换新角度 |
| 用户给 C-XXX 并说改、更新、替换 | 改文案 | 回查 copies，先展示修改版；确认后默认创建新版，不覆盖原文案 |
| 用户只说“选择第 N 条 / 选 4” | 选择确认 | 不要立刻写入；先判断上文任务，如果没有明确“入库”或“出图”，问用户下一步是入库、基于它出图，还是继续改文案 |
| 用户一次选择多条文案 | 多选确认 | 问这些文案是都入库、继续筛选，还是只选其中一条去出图；不要默认全部入库或默认第一条出图 |
| 用户给方向但缺渠道或形式 | 新建缺口 | 只问缺失项 |
| 用户粘文案样本并说类似、仿照、换说法 | 轻量扩文案 | 继承样本结构，换新角度；不要求补方向血缘 |
| 用户只说“出文案” | 信息缺失 | 问方向、渠道、图片形式 |

文案必须服从方向卡的字段 4/5/6：场景问题、差异化解法和奇效不能跑偏。如果想写的卖点不在方向里，不要强写，换一个切入角度。

用户未指定人群时，默认学生视角。学生视角不是强制第一人称，而是从学生真实场景、真实情绪和真实语言出发；语气可以活泼、有网感，但必须服务于当前功能和真实用户处境。文案内部按“场景问题 -> 惊艳解法 -> 场景奇效”组织：先让学生觉得“是我”，再用洋葱的具体产品动作让人好奇，最后给出可感知变化。不要把这个三段推理展示给用户。

多渠道/多形式请求先做合法组合过滤，不做简单笛卡尔积。应用商店和信息流可生成单图、双图、三图；学习机只支持单图。用户同时要求应用商店和学习机单图时，应用商店单图和学习机单图合并展示，展示为一组共用文案；如果用户要求两个渠道都入库，拆成 2 条 records，渠道分别写 `应用商店` 和 `学习机`。用户说“全渠道全形式各 2 套”时，按当前规则展示是 12 条；如果全部入库，Base 写 14 条。

新建多套文案时要有角度差异。3 套及以上至少覆盖 3 类切入角度；5 套时同一文案类型最多 2 套。不要把同一句钩子换几个近义词当多套。

## 反问原则

只问阻塞项：

- 缺方向时，问用户选方向、粘方向，或先去 `onion-direction` 生成。
- 缺渠道时，问信息流 / 应用商店 / 学习机 / 全部；不能因为上文示例或模型偏好默认成信息流。
- 缺图片形式时，问单图 / 双图 / 三图 / 全部；不能因为用户只说“5 条文案”就默认单图。若渠道是学习机，只提供单图。
- 缺数量时不问，按默认值。
- 扩文案样本结构清楚时不问，先输出。

有交互 UI 时一次性收集缺失项；没有 UI 时用一句话问清关键缺口。具体运行时适配见 `../../shared/references/runtime-adapters.md`。

## 硬约束

- 单图必须有 `主标题` 和 `副标题`。
- 双图必须有 `短句1` 和 `短句2`，`短句3` 留空。
- 三图必须有 `短句1`、`短句2`、`短句3`。
- 学习机只支持单图；应用商店双图/三图仍可用，且多图必须从同一个单图核心卖点延展。
- 生成多个渠道/形式时，必须先输出生成范围表，再按分组表格展示；编号全局唯一。
- 仅当用户同时要求应用商店和学习机单图时，应用商店单图和学习机单图合并展示；如果用户要求两个渠道都入库，写 Base 时拆成两条 records。用户只选共用文案但没说渠道时，先问入库应用商店、学习机，还是两个都入库。
- 不适用的字段不传空字符串；写入时让脚本用 `null` 或省略处理。
- 信息流避免“立即下载 / 限时优惠 / 快来抢 / 火爆开抢”等动词陷阱。
- 默认学生视角；用户未指定家长/老师/其他人群时，不写妈妈省心、家长减负、陪伴温暖等家长腔。
- 面向学生时要活泼、口语化、有强共鸣，但不能脱离功能事实；产品动作不清楚时先查知识库或追问，不能为了共鸣编功能。
- 每套文案都要让用户知道产品在做什么：优先出现 `洋葱`、`洋葱一拍`、`洋葱拍题精学`、`看不懂再问 AI`、`分步解析` 等产品动作；不写只有“高效学习/学习神器”的空话。
- 私教班手册中已有的提分、100%紧贴等表达可以作为业务口径吸收；不要自行扩张成包过、保证、唯一、第一等绝对化承诺。
- 广告主语默认是洋葱学园 APP 学生端，不把教师版、校园版、合作硬件或其他产品线写成主角。
- 迭代过程不写 Base；只有用户明确确认入库、或明确选择某条并要求进入下游出图时才写入。
- 用户只说“选择第 N 条 / 第二条 / 就用这个”但没有说入库或出图时，不要立刻写 Base；先问下一步是入库、基于它出图，还是继续改文案。
- 用户说“选择第 N 条 / 第二条 / 就用这个”并继续要求出图，视为确认采纳该文案：如果该文案还未入库，先只把被选中的文案写入 Base，拿到 C-XXX/record_id 后再进入 `onion-image`。
- 用户选择多条文案并要求出图时，必须问是每条各出图、还是只基于其中一条出图；不要默认第一条。
- 扩文案默认先展示再问是否入库，避免污染文案库。
- 改文案默认创建新版，不覆盖原文案；用户明确说废弃原文案时，先确认后把原文案状态改为 `废弃`。用户明确要求“覆盖原记录”时，说明当前不支持覆盖字段，只能创建新版并保留历史。

## 工具调用

用户给 D-XXX / C-XXX 但上下文没有对应内容时，先用 `../../shared/scripts/lookup_record.py --id D-XXX` 或 `--id C-XXX --follow-upstream` 回查。不要要求用户重复粘贴 Base 里已有内容。用户只粘贴临时文案样本时，不需要回查 Base。

用户确认入库后，用 `../../shared/scripts/write_record.py` 写入 copies 表。

每条记录用 `records[].fields` 表达，常用字段包括 `关联方向`、`渠道`、`图片形式`、`文案类型`、`主标题`、`副标题`、`短句1`、`短句2`、`短句3`、`状态=待用`。脚本会转换成当前 Base 写入需要的 `fields/rows` payload，并处理 dry-run、重试和 pending 队列。

不要传 `文案ID`、创建时间、创建人、最后更新时间；飞书会按当前 `--as user` 账号自动记录作者和时间。

如果文案关联了方向 record_id，写入成功后用 `../../shared/scripts/update_status.py` 把对应 directions 记录的 `状态` 改为 `已用`。如果只是用户临时粘贴方向内容，不更新上游状态。

如果用户明确要求废弃原文案，用 `../../shared/scripts/update_status.py` 把原 copies record_id 的 `状态` 改为 `废弃`；不要用 `write_record.py` 假装覆盖原记录。
