# onion-using

## 任务目标

这是洋葱投放插件的使用协议，不是中心业务执行器。它只帮助 Agent 判断该直接进入哪个 onion skill，以及在入口不明确时怎么分诊。

核心规则：明确任务直接进入对应原子 skill；歧义、跨边界、D/C/G ID、上传图用途不明、选择第 N 条时在本 skill 内完成分诊规则，再进入目标原子 skill。每个原子 skill 必须能独立整理 Input Envelope，不能假设某个中心分诊 skill 一定已经先运行。

## 入口判断

| 用户请求 | 推荐入口 |
|---|---|
| 明确出方向、扩方向、改方向 | `onion-direction` |
| 明确出文案、扩文案、改文案 | `onion-copy` |
| 明确用已确认文案 / C-ID / 临时文案出新图 | `onion-image` |
| 明确基于旧图 / G-ID / 爆款图扩同类或微调 | `onion-image-iterate` |
| 环境检查、Base 结构、pending、API key、图没入库 | `onion-help` |
| 不知道入口、只说选第 N 条、D/C/G ID 意图不明、上传图用途不明、跨方向/文案/图片边界 | 先按下方分诊规则判断，再进入目标原子 skill |

## 使用规则

- 首启环境门禁：任何 onion 请求开始时，先做轻量 readiness 判断。若 `~/.onion-ad/usage-state.json`、`~/.onion-ad/setup-status.json` 或 `~/.onion-ad/update-status.json` 不存在，先转 `onion-help` 运行 `setup_wizard.py ensure` 做安全本地配置和 24 小时缓存的插件更新检查；若平台/版本明显过期，或用户要写 Base / 生图 / 打开本地服务但环境未就绪，先环境检查。需要关闭自动更新时可设置 `ONION_PLUGIN_AUTO_UPDATE=0`。不要让用户跑到后面才发现缺 lark-cli、API key、Python 依赖或插件版本落后。
- 先判断入口，不要把所有 onion 请求都塞进额外中间层。
- 正确优先于少问。投放素材里的误判会污染 Base、浪费生图费用或生成不可用素材；只要影响入口、图片角色、写入对象、版位、父图组、是否使用竞品参考等关键判断，有一点不清楚就追问。
- 进入任何 onion skill 前，都按 `../../shared/references/input-envelope.md` 整理 Input Envelope。
- 如果当前 skill 发现越界，停止当前执行并转交目标 skill；不要在当前 skill 里勉强完成别的 skill 的工作。
- 不靠聊天记忆决定写 Base。方向/文案写入需要用户确认；图片写入需要选择页 `accepted_schemes` 和状态机准入。

## 分诊规则

### ID 分诊

- `D-XXX` 是方向锚点：扩/改方向进 `onion-direction`；出文案进 `onion-copy`；不能直接出图。用户说“D-XXX 出图”时，先转 `onion-copy` 生成或确认文案。
- `C-XXX` 是文案锚点：扩、改、换说法进 `onion-copy`；出图、生图、做图进 `onion-image`。
- `G-XXX` 是图组锚点：扩、换、微调、同款再来进 `onion-image-iterate`。
- 无 ID 但有临时文案：用户要出图时进 `onion-image`，不强制补方向或文案 ID。

### 选择语义

- 选择方向：用户说“选择方向 2 / 方向一 / 就用这个方向”时，进 `onion-direction`。方向入库后必须问下一步：出文案、继续改方向，还是先停。
- 选择文案：用户说“选择第 2 条 / 我选这个文案”但没说下一步时，进 `onion-copy` 问清是入库、基于它出图，还是继续改文案；不要擅自出图。
- 选择图片：用户打开选择页后，不接受“选 set1 / 选第 2 套”这类口头选择 set 来入库。必须回到 `onion-image` 或 `onion-image-iterate`，要求用户在选择页标注并提交；提交后用户只需回来说“好了/已提交”，Agent 读取本地 `image-selection-result.json`。
- 多选：选择多个方向或多个文案后要求下游生产时，必须问每个都继续，还是只基于其中一个继续；不要默认第一个。

### 上传图先判断角色

上传图先判断角色，不要因为“用户上传了图”就默认进 `onion-image-iterate`。
APP 截图、风格参考图或 IP 参考图都只是新图任务的参考素材，不是旧广告图锚点。
不要靠图片内容、品牌外观或“有没有文案”判断角色；同一张图可以是旧广告图、竞品参考、构图参考、IP 参考或 APP 截图。角色由用户意图决定。

| 上传图角色 | 目标 skill |
|---|---|
| 自有旧广告图锚点：照这张改、同款、扩同类、换文案、换角色、换 CTA | `onion-image-iterate`，可写父图组 |
| 竞品/外部参考图：参考作业帮等外部图的版式、构图、信息层级 | 参考素材；不写父图组，不复刻品牌/UI/文案 |
| 构图/版式参考：只参考构图或信息密度 | 当前新图或迭代任务的参考素材 |
| APP 截图、功能界面截图、拍题结果页截图 | 当前新图任务的参考素材，通常仍走 `onion-image` |
| 风格参考图：色调、构图、质感参考 | 当前新图任务的参考素材，通常仍走 `onion-image` |
| IP 参考图：新增角色或临时角色形象 | 当前新图任务的参考素材，通常仍走 `onion-image` |
| 用户只上传图但没说用途 | 先问：“这张图是我们旧广告图要改，还是竞品/外部参考、构图/风格参考、IP 参考，或 APP 截图？” |

## 常见越界

- `D-XXX 出图`：不是新图入口；先转 `onion-copy` 形成文案。
- `选第二条`：上下文不明时先判断是方向、文案还是图片方案。
- `选 set1 入库`：不能口头入库；转图片 skill 读取选择页 JSON。
- 上传图但没说明用途：先判断是旧广告图锚点，还是 APP 截图 / 风格图 / IP 参考。
- 上传作业帮等竞品/外部图：除非用户明确说这是自有旧素材，否则只能当竞品/外部参考图；不写父图组，不复刻品牌、UI、人物或文案。
- 旧图换角色、换文案、扩同类：转 `onion-image-iterate`，不要走新图配置。

## 硬约束

- 本 skill 不生成方向、不写文案、不生图、不写 Base，不调用图片 API。
- 本 skill 只做使用协议和分诊规则；原子 skill 必须自带入口门禁。
- Input Envelope 是内部交接对象，不是让用户填表。
