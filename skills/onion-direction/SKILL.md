# onion-direction

## 任务目标

把用户的产品功能、节点、人群或样本材料转成可进入投放生产链路的方向卡。方向卡是上游产物，下游会被 `onion-copy` 和 `onion-image` 引用。

成功的方向卡不是功能介绍，而是一个能被拍成 3 秒画面的投放切入点：具体人群、具体场景、具体痛点、差异化解法和可感知奇效都成立。

## 输入契约

必填输入：

- `功能`：从 `../../shared/knowledge/卖点库.md` 的功能矩阵理解真实产品能力；写入 Base 时对齐当前表选项（常见为拍题精学 / 同步课 / 总复习 / 学情报告 / 学习机 / 其他，新增功能先确认选项或映射）。
- `适配阶段`：日常学习 / P0-寒假 / P0-开学 / 期中前 / 期末前 / 暑期 / 其他。

可推断输入：

- `基准方向`：用户给 D-XXX / directions record_id 时先回查 Base；用户粘完整方向时直接用上下文。
- `目标人群`：用户给了年级、学习状态、场景或样本时可推断；否则可先用功能和节点生成一个合理细分。
- `卖点`：从功能、用户 brief、样本方向和 `../../shared/knowledge/卖点库.md` 推断；优先选择真实资料里出现过的卖点表达，再转成具体投放场景。
- `时间节点语义`：用户给日期、寒暑假、开学、期中、期末、月考、备考等信号时，推断学生当前状态和更适合的场景任务。
- `数量`：用户没说时默认 3 条；仿写没说时默认 2 条。

展示给用户和写入 Base 时，每条方向都必须具备 `素材方向`、`功能`、`卖点`、`目标人群`、`适配阶段`、三段方向字段和 `状态=待用`。不要传 `方向ID`、创建时间、创建人、最后更新时间；这些由飞书根据当前 `--as user` 账号自动记录。

## 推断原则

先判断模式：

| 信号 | 模式 | 处理 |
|---|---|---|
| 用户粘了完整方向并说类似、同款、仿照、换说法 | 仿写 | 保留结构和语气，换新角度；默认不主动入库 |
| 用户给 D-XXX 并说扩、再来、类似 | 扩方向 | 用 `../../shared/scripts/lookup_record.py` 回查原方向，保留核心人群/功能，可换卖点、场景或表达 |
| 用户给 D-XXX 并说改、更新、替换 | 改方向 | 回查原方向，先展示修改版；只有用户确认更新时才写 Base |
| 用户给功能、节点、人群、数量等明确 brief | 新建 | 不多问，直接生成 |
| 用户给调研/访谈/复盘材料 | 提炼 | 从材料里提炼场景、人群和痛点，再生成方向 |
| 用户只说“选择方向 N / 选第 N 条” | 选择确认 | 先把被选方向入库；入库后必须问下一步：是否基于该方向出文案、继续改方向，或先停在方向库 |
| 用户选择多个方向 | 多选确认 | 先确认这些方向都要入库；如要继续出文案，必须问是每个方向都出，还是只选其中一个方向出 |
| 用户只说“出方向” | 信息缺失 | 只问功能和适配阶段 |

完整 brief 不反问。用户没给目标人群但功能和节点足够明确时，可以自行选择一个细分人群，并在输出里自然体现，不要用确认流程拖慢创作。

多条方向要互补。新建创意时，任意两条至少在卖点、场景、人群、切入角度中有 2 维不同；3 条及以上至少覆盖 3 类切入角度，5 条及以上至少覆盖 4 类，同一角度最多 2 条。

时间节点要影响方向内容。不要只把“期末前/暑期/开学季”填进 `适配阶段`；字段 4/5/6 也要体现对应的学习状态、任务和解法。

## 反问原则

只问阻塞项：

- 缺功能时，问用户选哪个功能。
- 缺适配阶段时，问用户选哪个阶段。
- 缺数量时不问，按默认值。
- 仿写样本足够完整时不问，先产出。

有交互 UI 时一次性收集缺失选项；没有 UI 时用一句话问清关键缺口。具体运行时适配见 `../../shared/references/runtime-adapters.md`。

## 硬约束

- 方向卡必须使用 `references/字段与生成规则.md` 里的 6 字段语义。
- 候选方向展示时必须显式显示 `适配阶段`；不要只在内部字段里保留。
- `素材方向` 要像可投放主题，不要写成产品功能名。
- 字段 1 写具体场景里的具体问题；字段 2 写差异化解法；字段 3 写同一场景下的可感知变化。
- 不写包过、保证、提分 N 分、100%、唯一、第一等承诺词。
- 不写洋葱 APP 没有的能力。
- 广告主语默认是洋葱学园 APP 学生端，不把教师版、校园版、合作硬件或其他产品线写成主角。
- 迭代过程不写 Base。用户明确确认“OK / 满意 / 就这样 / 存进去”时写入；用户说“选择第 N 条 / 方向一 / 就用这个”视为确认采纳该方向，先只把被选中的方向写入 Base。
- 如果用户只是选择方向，没有明确说“出文案 / 继续生成文案”，入库后必须问下一步，不能直接结束也不能擅自出文案。
- 如果用户选择方向并明确要求出文案，入库后进入 `onion-copy`；若缺渠道或图片形式，由 `onion-copy` 继续反问。
- 如果用户一次选择多个方向并要求出文案，必须先问“每个方向各出一批，还是只基于其中一个方向出”，不要默认只用第一个。
- 仿写默认先展示再问是否入库，避免污染方向库。
- 出方向不依赖 `LAOZHANG_API_KEY`，不能因为缺老张 API key 阻塞。

## 按需 Reference

- 启动后读 `references/字段与生成规则.md`：字段语义、质量标准、内部检查表。
- 用户给日期、假期、开学、期中、期末、月考或备考信号时读 `references/时间节点策略.md`。
- 涉及产品边界或品牌说法时读 `../../shared/references/advertiser-subject.md`。
- 需要判断哪些业务知识可加载时读 `../../shared/references/business-knowledge.md`。
- 用户给 D-XXX / record_id 或要求断点续跑时读 `../../shared/references/record-lookup.md`。
- 判断是否仿写时读 `references/仿写模式.md`。
- 自查或需要历史经验时读 `references/经验参考.md`。
- 写 Base 前读 `../../shared/base_schema.md` 的 directions 表。
- 需要交互 UI 或异步反馈时读 `../../shared/references/runtime-adapters.md`。
- 用户反馈有规则沉淀价值时，参考 `../../shared/feedback_observation.md`；反馈失败不能阻塞主流程。

## 工具调用

用户给 D-XXX 但上下文没有方向内容时，先用 `../../shared/scripts/lookup_record.py --id D-XXX` 回查；拿到 record_id 后再决定扩写、展示或确认更新。

用户确认入库后，用 `../../shared/scripts/write_record.py` 写入 directions 表，table id 来自 `../../shared/base_schema.md` 或环境变量。

传给脚本的记录使用 `records[].fields` 结构，由脚本转换为当前 lark-cli 需要的 `fields/rows` payload。脚本负责 dry-run、重试和 pending 队列。

方向创建本身没有上游状态要更新；不要调用状态更新工具。
