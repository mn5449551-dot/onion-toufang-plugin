# 反馈观察指引

> 目标：把本轮真实、稳定、可复用的用户反馈沉淀到 `feedbacks` 表；不要把临时探索、口头试错、已经被用户改回来的偏好当成规则。

反馈收集是辅助能力，不是主任务。主 Skill 应保持业务流程顺畅：观察、不打断；能判断再写，判断不了就不写。

---

## 设计原则

| 原则 | 取舍 |
|---|---|
| 用户零负担 | 不弹满意度问询，不为了收集反馈打断方向、文案、出图流程 |
| 少写但写准 | 宁可漏掉弱信号，也不要把一次探索写成长期规则 |
| 先判断稳定性 | 看用户最终选择，而不是只看中途某一句否定 |
| 规则和偏好分开 | 可固化的业务边界写 `固定规则`；个体审美和单次吐槽写 `主观评价` |
| 写入走共享脚本 | Base 写入由 `shared/scripts/write_record.py` 负责，重试和 pending 兜底由脚本处理 |

默认不为反馈收集单独启动后台 subagent。只有运行环境明确支持、维护人明确要求，并且不会影响主流程时，才可以把“反馈归纳”拆出去；即便拆出去，Base 写入也仍应走共享脚本。

---

## 需要观察的信号

| 信号类型 | 典型表达 | 判断重点 |
|---|---|---|
| 连续负反馈 | 连续多轮说“不要 / 重出 / 不行 / 这个方向不对” | 最终是否仍废弃该对象 |
| 缺失抱怨 | “这个卖点你没写出来” | 是否是产品事实或卖点库缺口 |
| 错误指出 | “我说的是初一，不是高一” | 是否暴露输入理解、业务边界、字段推断问题 |
| 显式建议 | “下次目标人群写更具体” | 能否转成稳定生成原则 |
| 隐式偏好 | 用户反复手改同一字段 | 是否跨多轮出现，而非单次润色 |
| 合规问题 | “应用商店不让这样写/这样画” | 是否应升级为硬约束 |
| 业务越界 | “拍题精学没有这个功能” | 是否应进入业务知识或卖点库修正 |

观察笔记可以只存在当前上下文里，不需要每条都入库：

```text
[时间] [对象 D-XXX/C-XXX/G-XXX] [信号类型] 用户原话或概括
```

## 真反馈判断

| 场景 | 是否写入 | 理由 |
|---|---|---|
| 用户中途否定，最后又接受同一对象 | 不写 | 这是迭代探索，不是稳定反馈 |
| 用户连续否定某对象，最终废弃它 | 写 | 反馈和最终决策一致 |
| 用户指出产品事实错误 | 写 | 属于可复用的业务边界 |
| 用户只说“再来一版”“换个感觉” | 通常不写 | 信息量不足 |
| 用户给出明确改法 | 写 | 可沉淀为建议 |
| 用户当场改了，但没有表现出长期偏好 | 谨慎写 | 优先保留为主观评价，不升规则 |

## 字段映射

写入 `feedbacks` 表时只传业务字段，系统字段由飞书自动维护。

| 字段 | 内容 |
|---|---|
| `反馈对象类型` | `方向` / `文案` / `图组` / `Skill本身` |
| `被反馈对象ID` | `D-001` / `C-001` / `G-001` / `onion-direction` 等 |
| `反馈类型` | `固定规则` / `主观评价` |
| `反馈内容` | 用户原话优先；原话太长时用忠实概括 |
| `建议改法` | 能落地才写，不能落地可留空 |
| `处置状态` | 默认 `待审` |

不要传 `反馈ID`、`创建人`、`创建时间`、`最后更新时间`。这些字段由飞书根据当前 `lark-cli --as user` 身份自动填充，能记录真实作者和入库时间。

## 图片选择页反馈

图片流的反馈入口在标注页提交之后，而不是额外弹满意度问询。`image-selection-result.json` 是唯一依据：

| JSON 位置 | 写入方式 |
|---|---|
| `accepted_schemes` | 只用于 `image_groups` 入库，不当作反馈 |
| `rejected_schemes[].annotation.ruleFeedback` | 写 `反馈对象类型=图组`、`反馈类型=固定规则` |
| `rejected_schemes[].annotation.note` | 写 `反馈对象类型=图组`、`反馈类型=主观评价`；页面上叫“主观感受” |
| `rejected_schemes[].annotation.reason=跳过` 且没有具体文字 | 不写入 |
| `pending_scheme_ids` | 不写入，等待用户后续决定 |

未采纳的新图通常还没有 G-ID，`被反馈对象ID` 使用 `<request_id>:<set_id>` 作为临时锚点；如果选择结果里已有 `G-XXX` 或原图组 ID，则优先使用真实图组 ID。脚本会把渠道、版位、图片形式和问题图位写进 `建议改法` 作为分析上下文。

标注页的不采纳必须三选一：`固定规则` 要填写固定规则问题，`主观感受` 要填写具体主观反馈，`跳过反馈` 不写入 feedbacks。只要填写了固定规则或主观感受，`write_selection_feedback.py` 必须写入；如果 JSON 里不采纳但缺少理由或文字，状态机应停在 `invalid_selection_feedback`，不能继续打包或写图组。

---

## 写入方式

使用共享脚本写入，避免在 Skill 或 reference 里手拼 `lark-cli` 命令。

```bash
PLUGIN_ROOT=/path/to/onion-toufang-plugin
python3 "$PLUGIN_ROOT/shared/scripts/write_record.py" \
  --table-id "tblsPpNNcNH5KXoZ" \
  --records '[{"fields":{"反馈对象类型":"方向","被反馈对象ID":"D-007","反馈类型":"固定规则","反馈内容":"用户指出拍题精学没有语音讲解能力","建议改法":"从拍题精学卖点和生成规则中移除语音讲解表述","处置状态":"待审"}}]'
```

脚本会使用 `~/.onion-ad/.env` 的 Base 配置；写入失败时按共享 Base 操作逻辑重试，仍失败则进入 `~/.onion-ad/pending.jsonl`，后续由维护人或用户明确要求时运行补偿脚本处理。

图片选择页的固定规则 / 主观感受走专用脚本，避免 Agent 手工解析 JSON：

```bash
PLUGIN_ROOT=/path/to/onion-toufang-plugin
python3 "$PLUGIN_ROOT/skills/onion-image/scripts/write_selection_feedback.py" \
  --selection-result <output-dir>/image-selection-result.json
```

脚本会调用共享 `write_record.py` 写 `feedbacks`，并在同目录生成 `image-feedback-result.json`。后续 `image_workflow.py status` 会用这个文件判断反馈已处理，避免同一次标注重复写入。

---

## 不写入时的处理

没有显著反馈、或用户在标注页选择“跳过反馈”时直接继续主流程，不需要告诉用户“本轮没有反馈”。反馈系统的价值来自长期少量高质量样本，不来自每轮强行留痕。
