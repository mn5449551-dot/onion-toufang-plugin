# 洋葱投放插件手测用例

> 用法：建议每个大类开一个新对话测试，避免上文状态干扰。涉及 `D-XXX`、`C-XXX`、`G-XXX` 的用例，请替换成你飞书 Base 里真实存在的 ID；没有真实 ID 时先跑自然语言入口用例。

## 测试前检查

### T00｜环境检查

**用户输入**

```text
帮我检查一下洋葱投放插件环境，现在能跑了吗？
```

**期望行为**

- 进入 `onion-help`。
- 检查飞书 Base 配置、4 张表、pending 队列、`LAOZHANG_API_KEY`。
- 如果缺生图 key，明确说“不能付费生图”，但方向/文案/Base 查询仍可继续。

**通过标准**

- 不直接开始出方向、出文案或生图。
- 不等到 render 阶段才发现 key 缺失。

## 方向生成

### D01｜缺功能和卖点时只问关键输入

**用户输入**

```text
帮我出 3 个拍题精学方向
```

**期望行为**

- 进入 `onion-direction`。
- 只追问“功能是什么”和“核心卖点/利益点是什么”。
- 不追问“适配阶段”。

**通过标准**

- 没有直接生成方向。
- 没有要求用户填写一大堆字段。

### D02｜用户让 Agent 定卖点

**用户输入**

```text
功能是拍题精学，卖点你按当前节点推荐，帮我出 3 个方向
```

**期望行为**

- 根据当前时间节点推断 `适配阶段`。
- 从卖点库中选择合适卖点，不编造产品能力。
- 每个候选方向完整展示 6 个字段。

**通过标准**

候选排版应类似：

```text
方向 1｜...
功能：...
卖点：...
目标人群：...
适配阶段：...（按当前日期推断）

1 能解决用户在“具体哪个场景里的哪个问题”：
...

2 能带来什么不一样的“一听很惊艳”的解法？
...

3 因此带来了哪个场景下的什么“奇效”？
...
```

且不能出现：

- `推荐用图`
- `文案角度`
- `同上`
- `省略号`

### D03｜选择方向后只入库并询问下一步

**用户输入**

```text
我选方向 2
```

**期望行为**

- 只把方向 2 写入 `directions`。
- 写入后问：是否基于该方向生成文案、继续改方向，还是结束。

**通过标准**

- 不自动生成文案。
- 不把未选择的方向写入 Base。

## 文案生成

### C01｜选择方向后生成文案但缺渠道/形式

**用户输入**

```text
根据方向 2 帮我出 5 条文案
```

**期望行为**

- 进入 `onion-copy`。
- 如果上下文没有渠道和图片形式，必须追问。
- 不默认信息流，不默认单图。

**通过标准**

- 应先问类似“用于哪个渠道/版位，单图/双图/三图？”。
- 不直接输出信息流文案。

### C02｜明确渠道和形式后输出文案

**用户输入**

```text
基于刚才方向 2，出 5 条应用商店单图文案
```

**期望行为**

- 输出 5 条候选。
- 用户可读排版，不展示内部“角度/文案类型”。
- 文案要通顺，有产品主语和利益点，不出现“洋葱不懂还能继续问”这类病句。

**通过标准**

单图文案格式应类似：

```text
文案 1｜应用商店 · 单图
主标题：洋葱一拍，解析秒出
副标题：先看答案，不懂还能继续问
关联方向：D-XXX / 临时方向 / 无
```

### C03｜选择文案后先判断用户意图

**用户输入**

```text
我选第 2 条
```

**期望行为**

- 如果用户没有说下一步，先问是要入库、基于它出图，还是继续改文案。

**通过标准**

- 不直接入库。
- 不直接打开图片配置页。

## 新图生成 onion-image

### I01｜只给 D-XXX 不能直接出图

**用户输入**

```text
基于 D-XXX 帮我出图
```

**期望行为**

- 不进入 `onion-image` 生图流程。
- 说明方向 ID 不是生图锚点，需要先生成或确认文案。
- 转到 `onion-copy`。

**通过标准**

- 不打开 `/image-config`。
- 不写 prompt。
- 不调用 render。

### I02｜基于 C-XXX 出图必须先开配置页

**用户输入**

```text
基于 C-XXX 帮我出 2 套图
```

**期望行为**

- 回查文案。
- 启动图片配置页 `/image-config`。
- 配置页必须让用户明确选择版位，不默认选第一个版位。

**通过标准**

- 没有 `image-config-result.json` 时不能进入 prompt/render。
- 用户保存配置前不付费生图。

### I03｜多版位共用业务配置但分别出 prompt

**用户输入**

```text
用这条文案出图：洋葱一拍，解析秒出。应用商店 OPPO 横版大图和 vivo 搜索富媒体三图都要，2 套，IP 随机，字体参考启用。
```

**期望行为**

- 仍然先打开配置页确认。
- 版位可多选。
- 保存后每个版位单独生成 prompt 和 `render_size`。
- 随机 IP / 字体按每张图或每套图分别抽取，不因共用配置而固定成同一个。

**通过标准**

- 不让用户再单独选择比例。
- 配置结果里能看到多个 `placements[]`。
- 每个 placement 都带 `target_size`、`render_size`、KB 上限。

### I04｜需要真实 APP 界面截图时必须追问上传

**用户输入**

```text
帮我做一张信息流图，画面里手机上展示洋葱拍题解析页，文案是：拍一下，先看答案，不懂还能继续问。
```

**期望行为**

- 先开配置页。
- 如果配置或 prompt 判断需要可识别 APP/学习界面，必须提醒用户回到 Codex 上传真实截图。
- 用户没上传前不能 render。
- 用户不上传时，只能改成弱化/模糊屏幕内容。

**通过标准**

- 不编造拍题结果页、解析页、继续追问页 UI。
- `image_workflow.py status` 应处于 `needs_ui_reference_upload`。

### I04b｜上游文案形式和渠道锁定配置页可选版位

**用户输入**

```text
C-XXX 是信息流单图文案：主标题“拍一下，先看答案”，副标题“洋葱逐步讲解，哪步不懂问哪步”。帮我出图。
```

**期望行为**

- 仍然先打开配置页。
- 配置页只能提交信息流单图版位。
- 应用商店、学习机、双图、三图版位临时置灰并展示原因。
- 后端保存时也拒绝手工提交不匹配的 `placement_ids`。

**通过标准**

- `image-config-result.json` 的 `channel/categories` 是信息流。
- `image-config-result.json` 的 `image_form` 是单图。
- 不出现应用商店或学习机版位被写入配置结果。

### I05｜批量 20 套要分批写 prompt

**用户输入**

```text
用这条文案出 20 套应用商店单图：洋葱一拍，解析秒出。IP 随机，场景多样一点。
```

**期望行为**

- 先配置页确认。
- prompt 生成按 5-10 套一批推进。
- 保持多样性 ledger，避免 20 套都是同一 IP、同一场景、同一构图。

**通过标准**

- 不一次性粗糙吐 20 条 prompt 直接 render。
- 每批 render 前能看到 prompt 质量检查。

## 选择页、反馈和入库

### S01｜不能口头选择 set 入库

**用户输入**

```text
选 set1，帮我入库
```

**期望行为**

- 提醒必须以选择页提交的 `image-selection-result.json` 为准。
- 如果页面无法保存，应启动本地服务后重试保存，不要求用户粘贴 JSON。

**通过标准**

- 不因为一句“选 set1”就上传飞书 Base。

### S02｜标注反馈应写入 feedbacks

**操作步骤**

1. 在选择页采纳 `set1`。
2. 不采纳 `set2`，反馈类型选“固定规则”，填写：`应用商店图不能出现夸张提分承诺。`
3. 不采纳 `set3`，反馈类型选“主观感受”，填写：`人物表情不够可信，画面太硬广。`
4. 点击提交。
5. 回到 Codex 输入：

```text
我已经在选择页提交了，帮我入库
```

**期望行为**

- 先读取 `image-selection-result.json`。
- 发现 rejected 里有明确反馈，先运行 `write_selection_feedback.py`。
- 写出 `image-feedback-result.json`。
- 固定规则/主观感受进入 `feedbacks` 表。
- 只有 `accepted_schemes` 进入后续图片包和 `image_groups`。

**通过标准**

- `feedbacks` 表新增 2 条待审记录。
- `image_groups` 不包含 set2/set3。
- 选择“跳过反馈”不应写入反馈表。

### S03｜图片入库前必须打包本地 zip

**用户输入**

```text
选择页已经提交了，把采纳的图入库
```

**期望行为**

- 先运行 `package_accepted_images.py`。
- zip 只包含采纳图片。
- zip 外部生成 `<zip-stem>-manifest.json`。
- 再运行 `write_image_group.py --package-zip <zip_path>` 上传压缩图到飞书。

**通过标准**

最终回复必须包含：

- 飞书图组记录 ID / G-ID
- 可点击的本地图片包链接，使用脚本返回的实际绝对路径。
- 可点击的交付目录链接，使用脚本返回的实际输出目录。
- 外部 manifest 路径
- `image-write-result.json` 路径

zip 内命名应类似：

```text
set01_set1/set01_img01_1280x720_150kb.jpg
```

## 旧图迭代 onion-image-iterate

### G01｜G-XXX 扩同类走迭代，不走新图

**用户输入**

```text
G-XXX 这套效果不错，帮我扩 3 套同类图
```

**期望行为**

- 进入 `onion-image-iterate`。
- 回查原图组。
- 默认继承原图组的版位、图片形式、Logo、CTA、核心卖点。
- 打开继承型配置页，不从零配置。

**通过标准**

- 新图组写 `父图组`。
- 扩同类每套仍是完整图组；如果原图是三图，就输出多套三图，不拆成单图。

### G02｜上传图角色不清必须追问

**用户输入**

```text
参考这张图帮我再来几套
```

同时上传一张图片。

**期望行为**

- 不靠图片外观猜。
- 追问这张图是自有旧广告图、竞品/外部参考、构图/风格参考、IP 参考，还是 APP 截图。

**通过标准**

- 不直接写 `父图组`。
- 不直接进入 render。

### G03｜竞品参考图不能写父图组

**用户输入**

```text
这是一张作业帮的广告图，参考它的信息层级和构图，给洋葱做一套新的。
```

**期望行为**

- 把图片角色识别为 `competitor_reference` / `layout_reference`。
- 仍然按新图参考素材处理，不写 `父图组`。
- prompt 只抽象参考构图、节奏、信息层级，不复刻竞品品牌、UI、人物或文案。

**通过标准**

- 最终 image_groups 不应有 `父图组`。
- 不出现作业帮品牌或原文案。

### G04｜三图上传作为一套迭代

**用户输入**

```text
我上传的是同一组三图，整体感觉可以，帮我扩 3 套同类，文案换得更适合暑假。
```

同时上传三张图片。

**期望行为**

- 把三张图识别为同一基准图组，而不是三份独立配置。
- 使用一份继承型配置卡。
- 如果每张图改动不同，记录到 `per_image_notes`。

**通过标准**

- 不要求用户分别配置图1/图2/图3。
- 输出 3 套完整三图。

## 断点续跑

### R01｜已有 request_id 继续时先查状态

**用户输入**

```text
继续刚才 request_id 为 <request_id> 的生图流程
```

**期望行为**

- 先运行 `image_workflow.py status`。
- 根据 stage 决定下一步：
  - `needs_config`：只开配置页。
  - `needs_selection`：只开选择页。
  - `needs_feedback_write`：先写反馈。
  - `needs_package`：先打包。
  - `ready_to_write_base`：再写 Base。
  - `complete`：不重复写入。

**通过标准**

- 不凭聊天记忆跳步。
- 不重复上传同一批图片到 `image_groups`。
